# Agent 层级与沟通架构

## 整体架构图

```
用户 (Telegram)
    ↓
大内总管 (Steward) - 唯一用户接口
    ↓ 本地知识库
中书省 (Chancellery) - 决策起草
    ↓ 本地知识库
门下省 (Inspectorate) - 审核批准
    ↓ 本地知识库
尚书省 (State Affairs) - 执行调度
    ├─→ 本地知识库 → 六部
    └─→ GitHub → 地方项目

六部 (本地知识库通信)
├── 户部 (Revenue) - 资源监控
├── 工部 (Works) - 项目实施
│       ↓ spawn
│   ├── Builder - 编码实现
│   ├── Research - 技术调研
│   └── KO - 知识整理
├── 刑部 (Justice) - 运维监控
├── 兵部 (War) - 安全审查
├── 礼部 (Rites) - 知识库
└── 吏部 (Personnel) - KPI管理

地方 (GitHub通信)
├── FLOB2B - B2B平台项目
└── STD-Site - 独立站项目
```

## 关键原则

### 1. 单向上报
- 每个 Agent 只向直接上级汇报
- 不越级沟通
- 特殊情况（紧急）可临时越级，但需记录

### 2. 层级隔离
- 决策层（三省）不与执行末端直接沟通
- 六部不与地方项目直接沟通（通过尚书省协调）
- 地方项目独立运行，定期同步

### 3. 工具对应
| 通信类型 | 工具 | 场景 |
|----------|------|------|
| 用户沟通 | Telegram | 大内总管 ↔ 用户 |
| 中央内部 | 本地知识库 | 三省六部之间 |
| 中央-地方 | GitHub | 尚书省 ↔ 地方项目 |
| 任务派发 | spawn | 工部 ↔ Builder/Research/KO |

### 4. 信息流向
- 指令流：用户 → 大内总管 → 中书省 → 门下省 → 尚书省 → 六部/地方
- 汇报流：六部/地方 → 尚书省 → 大内总管 → 用户
- 信息流：所有节点记录到对应知识库

## 异常处理

### 紧急事件
1. 大内总管直接通知尚书省（跳过中书、门下）
2. 尚书省立即执行
3. 事后补充文档到知识库

### 跨级需求
1. 必须经尚书省协调
2. 六部不与地方直接沟通
3. 所有协调记录保存

### 地方独立决策
1. 日常事务可自行决定
2. 重大变更必须上报尚书省
3. 长期不汇报视为异常
