# AI Government 架构设计

## 核心理念

**三省六部制 + 多项目 × 多部署环境**

支持：
- **多国策**: 多个项目并行管理
- **多地方**: 多个部署环境（服务器/云/本地）
- **灵活组合**: 任何项目可部署到任何环境

## 三层架构

```
┌─────────────────────────────────────────────────────────────┐
│                    中央 - 行政中枢（全局）                     │
│  大内总管 → 中书省 → 门下省 → 尚书省 → 六部（全局职能）        │
└─────────────────────────────────────────────────────────────┘
                              ↓ GitHub
┌─────────────────────────────────────────────────────────────┐
│                    州 - 项目层级（管理层）                     │
│                    州牧（项目负责人）                          │
│         管理该项目的多个部署环境                               │
└─────────────────────────────────────────────────────────────┘
                              ↓ 管理
┌─────────────────────────────────────────────────────────────┐
│                   地方 - 部署层级（执行层）                    │
│              部署环境负责人 + 地方六部（在该环境执行）          │
└─────────────────────────────────────────────────────────────┘
```

## 三层详解

### 第一层：中央（全局管理）

只有一个中央，管理所有项目和所有部署环境的全局事务。

| 组件 | 职责 |
|------|------|
| **大内总管** | 用户唯一接口，奏折上报 |
| **中书省** | 起草国策（宏观项目规划）|
| **门下省** | 审核国策 |
| **尚书省** | 国策分解，下发给各州牧 |
| **六部** | 全局职能（全局资源、全局安全、全局知识等）|

### 第二层：州（项目层级）

每个项目有一个州牧（项目负责人）。

**州牧职责**：
- 管理该项目的整体规划
- 协调该项目的多个部署环境
- 接收中央国策，分解为项目任务
- 分配任务到各部署环境
- 汇总各环境进展，向中央汇报

### 第三层：地方（部署层级）

每个部署环境（服务器/云环境）有一套地方六部。

**部署环境类型**（模板提供）：
- AWS云环境
- 阿里云环境
- 私有部署环境
- Kubernetes集群
- 等等

**地方六部职责**：
- 在该具体环境中执行任务
- 向部署环境负责人汇报
- 向项目对应部门业务咨询

## 多项目 × 多环境 示例

### 场景：2项目 × 2环境

| | 阿里云生产 | AWS测试 |
|---|---|---|
| **B2B平台项目** | 州牧A管理该环境的地方六部 | 州牧A管理该环境的地方六部 |
| **电商站项目** | 州牧B管理该环境的地方六部 | 州牧B管理该环境的地方六部 |

### 完整架构图

```
中央
├── 尚书省
│   └── 下发国策
│       ├──→ 州牧A（B2B平台项目）
│       │       ├──→ 阿里云生产环境
│       │       │       ├── 地方工部（执行代码）
│       │       │       ├── 地方兵部（安全操作）
│       │       │       └── ...
│       │       └──→ AWS测试环境
│       │               ├── 地方工部（执行代码）
│       │               └── ...
│       └──→ 州牧B（电商站项目）
│               ├──→ 阿里云生产环境
│               └──→ AWS测试环境
│
└── 六部（全局）
    ├── 工部：技术指导（与各地方工部业务沟通）
    ├── 兵部：全局安全 + 对外操作
    └── ...
```

## 沟通渠道矩阵

### 纵向：行政线（汇报类）

```
地方六部 → 部署环境负责人 → 州牧 → 尚书省 → 大内总管 → 用户
```

### 横向：业务线（专业咨询）

```
地方工部 ↔ 项目工部 ↔ 中央工部（GitHub业务沟通）
地方兵部 ↔ 项目兵部 ↔ 中央兵部（GitHub业务沟通）
```

### 对外操作

```
所有环境的外部操作统一由 中央兵部 执行（SSH/API登录服务器等）
```

## 模板结构

```
templates/
├── projects/           # 项目类型模板（国策模板）
│   ├── web-platform/   # Web平台项目
│   ├── ecommerce/      # 电商项目
│   └── ai-service/     # AI服务项目
│
└── regions/            # 部署环境模板（地方模板）
    ├── cloud-aws/      # AWS云环境
    ├── cloud-ali/      # 阿里云环境
    └── on-premise/     # 私有部署环境
```

## 使用流程

### 1. 初始化中央（全局只需一次）

```bash
# 部署中央六部到 ~/.openclaw/central/
```

### 2. 创建新项目

```bash
# 基于模板创建项目
cp -r templates/projects/web-platform ~/.openclaw/projects/my-b2b/
# 配置项目信息，指定州牧
```

### 3. 添加部署环境

```bash
# 在项目下添加部署环境
cp -r templates/regions/cloud-ali ~/.openclaw/projects/my-b2b/prod-ali/
# 配置环境参数
```

### 4. 初始化环境

```bash
cd ~/.openclaw/projects/my-b2b/prod-ali
# 初始化地方六部
./init.sh
```

## 关键设计决策

### 为什么州牧管理多个环境？

一个项目通常需要多个环境（生产、测试、开发），由同一个州牧管理可以保持：
- 一致性：各环境配置一致
- 协调效率：跨环境任务协调
- 全局视野：了解项目整体状态

### 为什么地方六部在具体环境？

每个环境的执行细节不同（IP地址、资源配置、网络拓扑），需要本地化的：
- 执行能力：知道如何在该环境操作
- 监控能力：能访问该环境的指标
- 响应能力：快速处理该环境的问题

### 为什么中央兵部统一对外？

安全考虑：
- 统一入口：所有对外操作（SSH/API）都经过兵部
- 审计完整：所有操作记录在中央
- 权限控制：避免各地随意操作外部系统
