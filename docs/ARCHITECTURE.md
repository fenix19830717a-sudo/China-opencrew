# AI Government 架构设计

## 核心理念

**三省六部制是中国历史上最稳定、最成熟的政治架构，历经千年检验。**

从隋朝创立，经唐朝完善，一直沿用到清末，历时1300余年。

## 架构概览

### 中央 - 行政中枢

```
用户（皇帝）
    ↓ 奏折上报（OpenClaw人机交互）
大内总管（贴身太监）
    ↓ 本地共享知识库
中书省（决策起草）
    ↓ 本地共享知识库
门下省（审核批准）
    ↓ 本地共享知识库
尚书省（执行调度）
    ↓ 本地共享知识库
六部（具体执行，自己完成）
├── 户部（资源/财务）
├── 工部（工程/开发）自己编码
├── 刑部（运维/监控）自己排查
├── 兵部（安全/审查）自己审查
├── 礼部（知识/文档）自己整理
└── 吏部（人事/KPI）自己评估
```

### 地方 - 州郡

```
中央尚书省
    ↓ GitHub共享知识库
州牧（地方尚书省）
    ↓ 本地执行
地方六部（自己执行）
├── 地方户部
├── 地方工部 自己编码
├── 地方刑部 自己排查
├── 地方兵部 自己审查
├── 地方礼部 自己整理
└── 地方吏部 自己评估
```

## 关键设计决策

### 1. 六部自己执行（无Builder/Research/KO）

**为什么？**
- 简化架构，减少层级
- 六部就是执行者，不是管理者
- 工部自己写代码，不是spawn Builder

### 2. 地方复制中央架构

**为什么？**
- 保持一致性，降低理解成本
- 州牧 = 地方尚书省
- 地方六部 = 中央六部的地方版本

### 3. 奏折上报 = OpenClaw人机交互

**为什么？**
- 不限定具体渠道（Telegram/Slack/Discord都可）
- 核心是"人机交互"的概念
- 大内总管是唯一用户接口

### 4. 沟通机制分层

| 层级 | 渠道 | 场景 |
|------|------|------|
| 人机 | OpenClaw配置渠道 | 用户 ↔ 大内总管 |
| 中央内部 | 本地共享知识库 | 三省六部之间 |
| 中央-地方 | GitHub | 尚书省 ↔ 州牧 |
| 地方内部 | 本地共享知识库 | 州牧 ↔ 地方六部 |

## 六部详细职责

### 户部 - 资源财务
- 监控服务器、API额度、存储
- 成本核算和预算管理
- 资源预警和扩容建议

### 工部 - 工程开发
- 架构设计和技术选型
- 编码实现和单元测试
- 部署上线和代码审查
- **自己写代码，不spawn Builder**

### 刑部 - 运维监控
- 7x24系统监控
- 故障排查和修复
- 日志审计和告警
- **自己排查，不依赖Ops Agent**

### 兵部 - 安全审查
- 代码安全审查
- 权限管理和密钥轮换
- 漏洞扫描和风险评估
- **自己审查，不依赖安全Agent**

### 礼部 - 知识文档
- 知识库管理和维护
- 文档标准和质量把控
- 经验萃取和最佳实践
- **自己整理，不依赖KO**

### 吏部 - 人事绩效
- KPI定义和追踪
- 绩效评估和反馈
- 能力发展和培训
- **自己评估，不依赖外部Agent**

## 历史验证

三省六部制的核心优势：

1. **分权制衡** - 中书起草、门下审核、尚书执行，防止权力滥用
2. **专业化** - 六部分工明确，户工刑兵礼吏各司其职
3. **可扩展** - 中央-地方分级管理，郡县制可复制
4. **稳定性** - 人员更替不影响体系运行，铁打的营盘流水的兵

这些优势正好对应现代AI Agent协作的需求。

## 与OpenCrew的关系

本项目fork自OpenCrew，但进行了架构层面的重新设计：

| 特性 | OpenCrew | AI Government |
|------|----------|---------------|
| 架构 | 现代企业管理 | 古代三省六部制 |
| 角色 | CTO/Builder/Research | 六部尚书/州牧 |
| 执行 | spawn subagent | **六部自己执行** |
| 地方 | 简单项目Agent | **完整州牧+地方六部** |
| 沟通 | Slack为主 | **本地知识库+GitHub** |
| 人机 | Slack | **OpenClaw配置渠道** |
